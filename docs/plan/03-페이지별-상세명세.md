# Setflow — 페이지별 상세 명세서 (Page-level Detailed Spec)

> 본 문서는 `src/` 소스코드 100+ 페이지를 역할군(클러스터)별로 분석한 **페이지 단위 상세 명세**다.
> 각 페이지는 다음 8개 필드로 기술한다: ①라우트 ②유저 페르소나 ③목적 ④주요 UI/데이터 ⑤유저 액션 ⑥사용 상태/엔티티 ⑦백엔드 요구사항 ⑧UX 흐름.
> 상위 요구사항·백로그는 `01-요구사항정의서.md`, `02-기획서-토픽에픽태스크.md` 참조.

**클러스터 목차**
- [클러스터 1] 인증·온보딩·회원 운동 흐름 (10p)
- [클러스터 2] 회원 루틴·대시보드·커뮤니티·코칭·설정 (20p)
- [클러스터 3] 트레이너 워크스페이스 (25p)
- [클러스터 4] 헬스장(GYM) 운영 (30p)
- [클러스터 5] 플랫폼 운영 관리자 (18p)

# 클러스터 1 — 인증 · 온보딩 · 회원 운동 흐름


> 분석 대상: `src/pages/` 10개 페이지 + 공통 컴포넌트(`BottomNav`, `RestTimer`, `RouteGuard`) + `store.ts`(Zustand) + `types.ts`
> 스택: React + Vite + Zustand(`useGymStore`) + react-router-dom + framer-motion + date-fns + lucide-react + Tailwind(CSS 변수 테마)
> **현재 상태**: 모든 데이터는 인메모리 mock(`MOCK_SESSIONS`)이며 영속성/네트워크 없음. 인증은 실제 검증 없이 `completeOnboarding()`만 호출하고 통과. 아래 "백엔드 요구사항"은 프로덕션화를 위한 신규 설계.

## 공통 데이터 모델 (types.ts / store.ts)

```
MasterExercise { id, name, targetMuscle, icon }            // 마스터 운동 카탈로그 (현재 8개 하드코딩)
WorkoutSet     { id, setNo, type, weight, reps, completed, completedAt?, estimated1RM? }
SetType        = 'normal' | 'warmup' | 'drop' | 'fail'
WorkoutExercise{ id, baseExerciseId, name, targetMuscle, sets[] }
WorkoutSession { id, date(YYYY-MM-DD), exercises[], feedback? }
Routine        { id, name, description, color, exercises[] }
```

`useGymStore` 전역 상태(핵심):
- 온보딩/환경: `isFirstRun`, `weightUnit('kg'|'lb')`, `isDarkMode`, `userType('guest'|'general'|'trainer'|'gym'|'admin')`, `workoutGoals[]`, `userProfile{weight,height,age,gender}`
- 운동 데이터: `masterExercises[]`, `sessions: Record<date, WorkoutSession>`, `routines[]`, `activeRestTimer{duration, startedAt}|null`
- 액션: `completeOnboarding`, `setUserType`, `setWorkoutGoals`, `setUserProfile`, `logout`, `getSession`, `getOrCreateSession`, `addExercisesToSession`, `addSetToExercise`, `updateSet`, `removeSet`, `reorderExercises`, `startRestTimer`, `clearRestTimer`, `copySession`, `deleteSession`, `updateSession`
- **1RM 계산 로직(store 내장)**: `updateSet`에서 완료 시 `estimated1RM = round(weight * (1 + reps/30))` (Epley 공식). 세트를 방금 완료하면 `startRestTimer(120)` 자동 트리거.

### 백엔드 공통 스키마 (제안)

```sql
users(id, email, provider, provider_uid, created_at)
user_settings(user_id PK/FK, weight_unit, is_dark_mode, updated_at)
user_profiles(user_id PK/FK, weight, height, age, gender, updated_at)
user_goals(user_id FK, goal_id)                              -- loss|muscle|fitness|health
master_exercises(id, name, target_muscle, icon, is_custom, owner_user_id NULL)
workout_sessions(id, user_id FK, date, feedback, created_at, updated_at, UNIQUE(user_id,date))
workout_exercises(id, session_id FK, base_exercise_id FK, name, target_muscle, order_index)
workout_sets(id, exercise_id FK, set_no, type, weight, reps, completed, completed_at, estimated_1rm)
```

핵심 API: `POST /auth/social`, `POST /auth/login`, `PATCH /me/settings`, `PATCH /me/profile`, `PUT /me/goals`, `GET /sessions?from&to`, `GET /sessions/{date}`, `POST /sessions`, `POST /sessions/{date}/exercises`, `POST /exercises/{id}/sets`, `PATCH /sets/{id}`, `DELETE /sets/{id}`, `PATCH /sessions/{date}/reorder`, `POST /sessions/copy`, `DELETE /sessions/{date}`, `GET /master-exercises`.

## 1. SplashPage — 온보딩 시작 / 환경 설정

1. **페이지명 / 라우트**: SplashPage / `/splash`
2. **유저 페르소나**: 앱을 처음 실행했고 WelcomePage에서 "일반 회원"을 선택해 넘어온 신규 사용자. 아직 계정이 없고 단위·테마부터 정하려는 상태.
3. **목적**: 무게 단위(kg/lb)와 테마(라이트/다크)를 선택해 온보딩의 첫 환경 설정을 마친다.
4. **주요 UI / 표시 데이터**:
   - 상단 "뒤로 가기" 헤더(sticky), 🚀 브랜드 로고 + "Setflow" + 슬로건("운동 흐름은 그대로, 성장은 데이터로")
   - 단위 선택 카드: `kg` / `lb` 토글 버튼
   - 테마 선택 카드: `라이트`(Sun) / `다크`(Moon) 토글 버튼
   - 하단 "저장" CTA, 그 아래 "이미 계정이 있으신가요? 로그인" 링크
5. **유저 액션·인터랙션**:
   - `setUnit('kg'|'lb')`, `setIsDark(true|false)` 로컬 state 토글
   - 다크 토글 시 `useEffect`로 `document.documentElement.classList` add/remove `dark` (즉시 프리뷰)
   - "저장" → `handleComplete()`: `useGymStore.setState({isDarkMode})` 후 `navigate('/onboarding/goals', {state:{unit,isDark}})`
   - "뒤로 가기" → `/welcome`, "로그인" → `/auth/login`
6. **사용 store 상태 / mock 엔티티**:
   - 읽기: `isFirstRun` (false면 `<Navigate to="/calendar">`로 즉시 리다이렉트)
   - 쓰기: `isDarkMode`(setState 직접). unit/isDark는 store에 저장하지 않고 라우터 `location.state`로 다음 단계에 전달(주의: 여기선 `completeOnboarding` 미호출).
7. **백엔드 요구사항**:
   - 로컬 우선. 로그인 후 서버 반영: `PATCH /me/settings { weight_unit, is_dark_mode }` → `user_settings(user_id, weight_unit, is_dark_mode)`
8. **UX 흐름**: `/welcome`(일반 회원 선택) → **/splash** → 단위·테마 선택 → 저장 → `/onboarding/goals` (또는 로그인 링크 → `/auth/login`, 뒤로 → `/welcome`)

## 2. WelcomePage — 사용자 유형 선택 진입점

1. **페이지명 / 라우트**: WelcomePage / `/welcome`
2. **유저 페르소나**: 앱 최초 실행(`isFirstRun=true`)으로 RouteGuard에 의해 이 화면으로 유도된 사용자. 본인이 일반 회원인지, 트레이너인지, 헬스장 운영자인지 정체성을 고르는 단계.
3. **목적**: 3가지 사용자 유형 중 하나를 선택해 유형별 온보딩 분기로 진입한다.
4. **주요 UI / 표시 데이터**:
   - 🚀 로고 + "Setflow" + "환영합니다! 어떤 사용자로 시작하시겠어요?"
   - 카드 3개: 일반 회원(User, "나만의 운동 기록 관리") / 트레이너(Dumbbell, "회원 관리 및 수익 창출") / 헬스장·센터장(Building2, "소속 트레이너 및 매출 관리")
5. **유저 액션·인터랙션**:
   - 일반 회원 → `navigate('/splash')`
   - 트레이너 → `navigate('/trainer/onboarding/register')`
   - 헬스장 → `navigate('/gym/onboarding/register')`
   - (주의) 이 페이지 자체는 `setUserType`을 호출하지 않음 — 유형 확정은 각 온보딩/로그인 이후로 위임됨
6. **사용 store 상태 / mock 엔티티**: 없음(순수 네비게이션). 유형은 이후 단계에서 `setUserType`으로 확정.
7. **백엔드 요구사항**: 없음(정적 라우팅). 선택된 유형은 최종 가입 시 `users.role` 또는 별도 `account_type`에 저장.
8. **UX 흐름**: (앱 최초 실행) RouteGuard → **/welcome** → [일반]→`/splash` / [트레이너]→`/trainer/onboarding/register` / [헬스장]→`/gym/onboarding/register`

## 3. LoginPage — 로그인 (소셜 + 이메일)

1. **페이지명 / 라우트**: LoginPage / `/auth/login`
2. **유저 페르소나**: 이미 계정이 있는 재방문 사용자. Splash의 "이미 계정이 있으신가요?"를 눌러 들어와 기존 기록에 다시 접근하려 한다.
3. **목적**: 소셜 또는 이메일/비밀번호로 로그인해 캘린더(메인 앱)로 진입한다.
4. **주요 UI / 표시 데이터**:
   - 뒤로가기 헤더, 👋 로고 + "환영해요!" + 서브카피
   - 소셜 버튼 3개: 카카오(노란 배경 커스텀 아이콘) / 구글 / Apple
   - 구분선 "또는 이메일로 로그인"
   - 폼: 이메일 input, 비밀번호 input(show/hide 토글 Eye/EyeOff), "비밀번호를 잊으셨나요?"(미구현), "이메일로 로그인" submit
   - 하단 "아직 계정이 없으신가요? 회원가입"(버튼만, 핸들러 미연결)
5. **유저 액션·인터랙션**:
   - `email`/`password` 로컬 state, `showPassword` 토글
   - `handleLogin(e)`: `email && password` 없으면 무동작(return), 있으면 `completeOnboarding('kg', false)` 후 `navigate('/calendar', {replace:true})`
   - `handleSocialLogin()`: 검증 없이 `completeOnboarding('kg', false)` + `/calendar`
   - submit 버튼 disabled 스타일: 이메일·비번 둘 다 있을 때만 활성 색상
6. **사용 store 상태 / mock 엔티티**:
   - 쓰기: `completeOnboarding(unit, isDarkMode)` → `isFirstRun=false, weightUnit, isDarkMode` 설정 (실제 자격증명 검증/저장 없음)
   - **주의**: 로그인이 항상 `'kg', false`로 하드코딩되어 기존 사용자 설정을 덮어씀. `userType`도 설정 안 함(기본 guest 유지).
7. **백엔드 요구사항**:
   - `POST /auth/login { email, password }` → JWT/세션, 실패 시 401
   - `POST /auth/social { provider: kakao|google|apple, id_token }` → OAuth 검증 후 upsert `users(provider, provider_uid)`
   - `POST /auth/password/reset` (비밀번호 재설정)
   - 로그인 성공 시 `GET /me/settings`, `GET /me/profile`로 실제 unit/theme 복원(하드코딩 제거 필요)
8. **UX 흐름**: `/splash`(로그인 링크) → **/auth/login** → [소셜/이메일 로그인 성공] → `/calendar` (뒤로 → 이전 화면)

## 4. GoalsPage — 운동 목표 선택 (온보딩 2/4)

1. **페이지명 / 라우트**: GoalsPage / `/onboarding/goals`
2. **유저 페르소나**: Splash에서 단위·테마를 정하고 넘어온 신규 회원. 자신의 운동 동기를 앱에 알려 맞춤 경험을 기대하는 상태.
3. **목적**: 대표 운동 목표를 최대 2개까지 선택한다.
4. **주요 UI / 표시 데이터**:
   - 뒤로가기 + "건너뛰기" 헤더, 제목 "어떤 목표로 운동하시나요?", "최대 2개까지 선택"
   - 목표 카드 4개(GOALS 상수): 체중 감량🔥(loss) / 근육 증가💪(muscle) / 체력 향상🏃(fitness) / 건강 유지🌱(health) — 각 emoji + title + desc
   - 하단 "저장" CTA
5. **유저 액션·인터랙션**:
   - `toggleGoal(id)`: 선택/해제, 2개 초과 선택 불가(3번째부터 나머지 카드 `disabled` + opacity)
   - "저장" `handleNext()`: `setWorkoutGoals(selectedGoals)` 후 `navigate('/onboarding/body', {state:{unit,isDark}})`
   - "건너뛰기" `handleSkip()`: 저장 없이 `/onboarding/body`
   - 저장 버튼은 최소 1개 선택 시 활성
6. **사용 store 상태 / mock 엔티티**:
   - 읽기: `location.state`의 `{unit,isDark}` (없으면 기본 `kg/false`)
   - 쓰기: `setWorkoutGoals(string[])` → `workoutGoals`
7. **백엔드 요구사항**:
   - `PUT /me/goals { goals: string[] }` → `user_goals(user_id, goal_id)` (goal_id ∈ loss|muscle|fitness|health, 최대 2개 검증)
8. **UX 흐름**: `/splash` → **/onboarding/goals** → 목표 0~2개 선택 → 저장/건너뛰기 → `/onboarding/body` (뒤로 → `/splash`)

## 5. BodyProfilePage — 신체 정보 입력 (온보딩 3/4)

1. **페이지명 / 라우트**: BodyProfilePage / `/onboarding/body`
2. **유저 페르소나**: 목표를 정한 신규 회원. 체성분 리포트/데이터 분석을 위해 신체 수치를 제공할지 결정하는 단계(나중에 입력 가능 안내 존재).
3. **목적**: 나이·성별·키·체중을 입력해 개인화 분석 기반 데이터를 확보한다.
4. **주요 UI / 표시 데이터**:
   - 뒤로가기 + "건너뛰기" 헤더, 제목 "현재 신체 정보를 입력해주세요" + "정확한 데이터 분석/체성분 리포트를 위해 필요" 안내
   - 입력: 나이(number, 단위 "세") / 성별(남성·여성·기타 3버튼) / 키(number, "cm") / 체중(number, 단위=선택한 `unit`)
   - 하단 "저장" CTA
5. **유저 액션·인터랙션**:
   - `weight/height/age` 텍스트 input, `gender` 버튼 선택('M'|'F'|'O')
   - `isFormValid` = 4개 중 하나라도 입력되면 true → 저장 활성
   - "저장" `handleNext()`: `setUserProfile({weight?,height?,age?,gender})` (빈값은 undefined) 후 `navigate('/onboarding/auth', {state:{unit,isDark}})`
   - "건너뛰기" `handleSkip()`: 저장 없이 `/onboarding/auth`
6. **사용 store 상태 / mock 엔티티**:
   - 읽기: `location.state` `{unit,isDark}` (체중 단위 표시에 사용)
   - 쓰기: `setUserProfile(Partial<{weight,height,age,gender}>)` → `userProfile` 병합
7. **백엔드 요구사항**:
   - `PATCH /me/profile { weight, height, age, gender }` → `user_profiles(user_id, weight, height, age, gender)`
   - 향후 체성분 리포트용 시계열: `body_metrics(id, user_id, date, weight, body_fat, muscle_mass)`
8. **UX 흐름**: `/onboarding/goals` → **/onboarding/body** → 신체정보 입력/건너뛰기 → `/onboarding/auth` (뒤로 → `/onboarding/goals`)

## 6. AuthSignupPage — 가입 유도 / 게스트 시작 (온보딩 4/4)

1. **페이지명 / 라우트**: AuthSignupPage / `/onboarding/auth`
2. **유저 페르소나**: 온보딩 마지막 단계의 신규 회원. 클라우드 백업·코칭 등 전체 기능을 위해 가입할지, 우선 기기 로컬로만 시작할지 갈림길에 선 상태.
3. **목적**: 소셜 회원가입을 유도하되, 게스트(기기 저장) 시작 옵션도 제공하며 온보딩을 완료한다.
4. **주요 UI / 표시 데이터**:
   - 뒤로가기 헤더, 제목 "가입하고 내 기록 안전하게 보관하기" + "클라우드 백업/코칭 위해 무료 가입 필요"
   - 소셜 가입 버튼 3개(카카오/구글/Apple), "이메일로 가입하기" 링크
   - 하단 구분선 + "우선 기기에만 저장할까요?" + "가입 없이 바로 시작" 버튼
5. **유저 액션·인터랙션**:
   - `handleSocialSignup()`: (실제 OAuth 미구현) `completeOnboarding(unit, isDark)` 후 `/calendar` replace
   - `handleStartWithoutAuth()`: `completeOnboarding(unit, isDark)` 후 `/calendar` replace (게스트)
   - "이메일로 가입하기" → `navigate('/auth/signup')`
6. **사용 store 상태 / mock 엔티티**:

### B. RestTimer (`components/RestTimer.tsx`)
- **역할**: `activeRestTimer` 존재 시 하단 고정 카운트다운 카드. 진행바(progress %), `mm:ss`, `+15`초, 닫기(X→`clearRestTimer`). 0 도달 시 빨강 pulse(알람 미구현).
- **상태**: 읽기 `activeRestTimer{duration,startedAt}`, 쓰기 `clearRestTimer`. 로컬 `timeLeft`/`isPaused`(pause UI 버튼은 미노출, Play/Pause/RotateCcw import만).
- **주의**: `startedAt` 기반 절대시간 복원 없이 `setInterval` 카운트 → 리마운트/백그라운드 시 부정확. 프로덕션은 `startedAt`+`duration`으로 남은시간 재계산 필요.
- **백엔드**: 불필요(클라이언트). 다기기 동기화 원하면 `activeRestTimer`를 서버/로컬스토리지 영속화.

### C. RouteGuard (`components/RouteGuard.tsx`)
- **역할**: 라우팅 가드(Outlet). `isFirstRun`/`userType` 기반 접근 제어.
  - `isFirstRun && !public` → `/welcome`
  - `!isFirstRun && public` → 유형별 홈(trainer→`/trainer/home`, gym→`/gym/home`, else→`/calendar`)
  - `!isFirstRun && '/'` → 동일 유형별 홈 리다이렉트

# 클러스터 3 — 트레이너 워크스페이스

- `types.ts`: `WorkoutSet`(setNo,type,weight,reps,completed,estimated1RM), `WorkoutExercise`(baseExerciseId,name,targetMuscle,sets[]), `WorkoutSession`(date,exercises[],feedback?), `Routine`(name,description,color,exercises[]), `MasterExercise`(name,targetMuscle,icon).

# 클러스터 5 — 플랫폼 운영 관리자


> 대상 역할군: **플랫폼 운영 관리자 (Platform Admin)**
> 분석 대상: `src/pages/Admin*.tsx` 18개 페이지
> 참고 파일: `src/store.ts`, `src/types.ts`, `src/App.tsx`

## 공통 사항 (전 페이지 공통)

- **Store 미연동:** 18개 관리자 페이지 전부 `useGymStore`(store.ts)를 **전혀 사용하지 않는다.** store.ts는 개인 운동/헬스장(트레이너 초대) 도메인 전용이며, 관리자 도메인 엔티티(회원/심사큐/신고/정산/시스템설정)는 store에 정의조차 되어 있지 않다. 모든 데이터는 각 컴포넌트 내부의 로컬 `useState` 하드코딩 mock이며, 새로고침 시 초기화된다.
- **영속성 없음:** 승인/반려/제재/정산/저장 등 모든 쓰기 액션은 `setState`로 로컬 배열을 필터링하거나, `setTimeout` + `alert()` / `window.confirm()`으로 성공을 흉내낼 뿐 API 호출·DB 저장이 없다.
- **인증/인가 미구현:** 관리자 계정 판별, 권한 검사, 처리자(admin id) 식별 로직이 없다. 처리자 필드는 `'Admin (admin1)'` 같은 하드코딩 문자열.
- **공통 UI 패턴:** 상단 sticky 헤더(뒤로가기 + 타이틀), `framer-motion` 애니메이션, 카드형 리스트, 빈 상태(empty state) 플레이스홀더, SLA D-day 뱃지, confirm/alert 기반 처리.
- **백엔드 공통 요구:** 모든 관리자 액션은 `admin_audit_logs` 감사로그 테이블에 (actor_admin_id, action, target_type, target_id, before, after, created_at)로 남아야 하며, 처리 결과를 대상 유저/트레이너에게 알림(푸시/이메일/앱내)으로 발송하는 파이프라인이 필요하다.

## 18. AdminSystemLogsPage — 서버 상태 & 로그

1. **페이지명 / 라우트**
   서버 상태 & 로그 📡 / `/admin/system/logs`

2. **유저 페르소나**
   서버 업타임(SLA)과 장애/에러 로그를 모니터링하고 장애 알림을 관리하는 플랫폼 인프라/DevOps 담당 운영자.

3. **목적**
   기간별 서버 업타임(SLA 목표 대비)과 주요 장애·에러 로그를 조회하고 SLA 미달 알림을 토글.

4. **주요 UI / 표시 데이터**
   - 기간 필터 탭(24시간/7일/30일)
   - 업타임 현황 카드: 현재 업타임(99.8%) vs 목표 SLA(99.5%), 24구간 막대 차트(정상/지연/장애 색상), 장애 알림 ON/OFF 토글
   - 주요 장애/에러 로그 리스트: 레벨(error/warn), 서비스명, 타임스탬프, 메시지

5. **유저 액션·인터랙션**
   - 기간 탭 전환(현재 데이터는 정적)
   - 장애 알림 토글 → 상태 반전 + alert

6. **사용 store 상태 / mock 엔티티**
   - store: 없음
   - 로컬 상태 `period`, `alertsEnabled`, 상수 `currentUptime/targetSLA`, `uptimeBars[]`, `logs[]{ id, timestamp, level, service, message }`

7. **백엔드 요구사항**
   - `GET /admin/system/uptime?period=1d|7d|30d` — 모니터링 시스템(Prometheus/Datadog/StatusPage) 연동 집계
   - `GET /admin/system/logs?period=&level=` — 로그 수집기(ELK/CloudWatch) 조회
   - 테이블/스토어 `system_incidents(id, service, level, started_at, duration, message)`, `uptime_metrics(interval, status)`
   - `PUT /admin/config/alerts` — SLA 미달(99.5% 미만) 알림 on/off, 임계값 설정
   - 알림 발송 채널(Slack/이메일/PagerDuty) 연동

8. **UX 흐름**
   진입 → 기간 선택 → 업타임/SLA 확인 → 장애 구간·에러 로그 검토 → 알림 ON/OFF 토글
